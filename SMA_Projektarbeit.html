1. Einleitung

Ihr Chef ist von Chatbots begeistert und möchte auch ein Chatsystem in das eigene Produkt-Portfolio aufnehmen.

Als studentisches Team haben Sie die Aufgabe, einen lauffähigen Prototypen für ein Chatbot-Basissystem zu entwickeln. Das Basissystem dient als Laufzeitumgebung für eigene Chatbots. Das System sollte in der Lage sein, neue Chatbots einfach hinzuzufügen und nahtlos in das Gesamtsystem zu integrieren.

Erstellen Sie eine detaillierte Architekturbeschreibung ihres Systems, einschließlich der Schnittstellen zwischen den Modulen, der Datenbank, den Chatbots und die über die Chatsbots verfügbaren externen Systeme. Implementieren Sie dann ein Minimum Viable Product (MVP) Ihres Systems, das die grundlegenden Funktionen umsetzt und demonstriert, wie neue Chatbots hinzugefügt werden können.

Zudem sind für die Demonstration drei einfache technische Chatbots zu entwickeln.

2. Eigene Bots

2.1. Wetter-Bot

Ein Bot, welcher Informationen über das aktuelle Wetter und Wettervorhersagen geben kann.

2.1.1. Aktuelles Wetter

Der Bot kann zu aktuellen Wetter gefragt werden.

Eine beispielhafter Chatverlauf ist: 

user: @wetterbot wie ist das Wetter in Bielefeld?

wetterbot: In Bielefeld ist es aktuell
    - Temperatur: 10ºC
    - Windgeschwindigkeit: 5.6 m/s
    - Niederschlag: 0.0mm/h
    - ...

2.1.2. Vorhersage

Auch die Vorhersage des Wetters soll der Bot beantworten können.

Eine beispielhafter Chatverlauf ist: 

user: @wetterbot wie wird das Wetter in Bielefeld?

wetterbot: In Bielefeld ist die Vorhersage wie folgt:
    - Freitag, 31.03.2023 - 12ºC/4ºC, leichter Regen, leichte Brise
    - Samstag, 01.04.2023 - 14ºC/8ºC, kein Regen, kein Wind
    - ...

2.3. Übersetzer-Bot

Ein Bot, welcher einen Text in eine andere Sprache übersetzt.

2.3.1. Beispiel

user: @translatebot übersetze folgende Nachricht ins deutsche: A bird in the hand is worth two in the bush.

translatebot: Der Spatz in der Hand ist besser als die Taube auf dem Dach.

2.3.2. Datenquelle

Als Datenquelle kann z.B. die API von Deepl genutzt werden. Hier sind 500.000 Zeichen / Monat kostenlos

  • www.deepl.com Die Antwort wird ebenso als JSON zurückgegeben.
curl -X POST 'https://api-free.deepl.com/v2/translate' \
    -H 'Authorization: DeepL-Auth-Key 21947c34-929d-ad82-932b-5747f7ba2f31:fx' \
    -d 'text=A bird in the hand is worth two in the bush.' \
    -d 'target_lang=DE'

{"translations":[{"detected_source_language":"EN","text":"Der Spatz in der Hand ist besser als die Taube auf dem Dach."}]}

3. Funktionale Anforderungen

3.1. Bedienung / TUI / GUI

Das System ist über die Console per Textbefehle zu bedienen und die Textausgabe erfolgt ebenfalls über die Console.

Es soll im späteren Versionen auf ein anderes Frontend umgestellt werden. Daher muss eine Trennung zwischen dem System und der Bedienoberfläche konzipiert werden, die einen Wechsel auf ein anderes Frontend einfach ermöglicht.

3.1.1. Variante A

Ein Bot wird über ein @<name> tag angesprochen 

user: @<botname> <Befehl oder Frage>
bot X:  <Antwort>

3.1.2. Variante B

Ebenso ist es möglich, dass der Benutzer den Bot aktiviert und der Bot die Optionen vorgibt: 

user: @translatebot

translatebot: In welche Sprache soll ich übersetzen? Bitte zweistelligen Code eingeben:
    - [En]glisch,
    - [De]utsch,
    - ...

user: DE

translatebot: Bitte den zu übersetzenden Text eingeben:

user: A bird in the hand is worth two in the bush.

translatebot: Der Spatz in der Hand ist besser als die Taube auf dem Dach.

user: quit

translatebot: Bye!

3.1.3. Variante C

Analyse aus dem Text: 

user: Wie ist das Wetter in Bielefeld?

Wetterbot: Es regnet in Bielefeld

user: Was ist Bielefeld

Wikibot: ich habe folgendes zu Bielefeld gefunden:

  - Bielefeld ist eine Stadt
  - Arminia Bielefeld

3.1.4. Variante D

Eine eigene Idee für die Ansprache eines Bots ist ebenfalls möglich.

3.2. Benutzer

3.2.1. Benutzer

Das System muss mehrere Benutzer unterstützen. Eine zeitgleiche Verwendung des Systems ist nicht erforderlich.

3.2.2. Login

Der Benutzer muss sich beim Starten am System mit einem Benutzernamen und Passwort anmelden.

3.2.3. Benutzerverwaltung

Eine Benutzerverwaltung ist nicht erforderlich. Alle Benutzer können vorab im System definiert werden.

3.3. Persistenz

Das System sollte eine Datenbankanbindung haben, in der alle Chat-Nachrichten gespeichert werden. Die Datenbankanbindung sollte skalierbar und austauschbar sein.

3.3.1. Datenbank

Die Nachrichten können in diesem Prototypen in einer lokalen Datenbank (z.B. SQLite) gehalten werden. Für den späteren Betrieb ist aber eine performantere Datenbank angedacht. Ein einfacher Wechsel der Datenbank muss gegeben sein.

3.3.2. Chatverlauf

Wenn sich ein Benutzer anmeldet, bekommt er seinen Chatverlauf angezeigt (maximal 100 Nachrichten).

3.4. Hinzufügen von Chatbots

3.4.1. Initiales Laden

Es soll eine API / Schnittstelle bereitgestellt werden, über die Entwickler einfach neue Chatbots in das System laden können.

Diese Funktion ist für Entwickler gedacht und eine erforderliche Kompilation des Quellcodes ist hierfür in Ordnung.

3.4.2. Aktivieren von Bots

Um Resourcen zu sparen, sollen nicht immer alle Bots automatisch aktiviert werden.

Dem Benutzer muss es daher möglich sein, sich eine Liste verfügbarer Bots anzeigen zu lassen: 

<user>: list bots
system: Available bots:
    1) Weather bot (available)
    2) Translation bot (available)
    3) Wiki bot (enabled)

und diese enstprechend zu aktivieren und deaktivieren: 

user: activate bot 1
system: bot @weatherbot activated

user: deactivate bot 3
system: bot @wiki deactivated

4. Nicht-Funktionale Anforderungen

4.1. Programmiersprache

Java ist gewünscht, es kann aber davon abgewichen werden.

4.2. Sparchverständlichkeit

Die Bots müssen kein Textverständnis haben, eine Pattern-basierte Text analyse ist ausreichend. Ebenso ist es i.O., wenn der Bot Auswahloptionen vorgibt um somit die Textverarbeitung zu erleichtern.

Es wird die Textanalyse nicht bewertet! Wohl aber der Ansatz, wie die Texte analysiert werden. Auch hier zählen die Prinzipien guter Softwareentwicklung.

4.3. Modularität

Das System sollte aus verschiedenen Modulen bestehen, die unabhängig voneinander entwickelt und getestet werden können. Dadurch kann eine effiziente und verteilte Entwicklung gewährleistet werden.

4.4. Skalierbarkeit

Das System sollte so gestaltet sein, dass es problemlos erweitert werden kann, um neue Funktionen oder Chatbots hinzuzufügen.

6. Dokumentenstruktur

Eine gute Softwaredokumentation sollte eine umfassende, klare und präzise Beschreibung der Software liefern. Der Fokus liegt bei dieser Arbeit auf die Entwicklerdokumentation.

Da die Dokumentation in Markdown erstellt werden soll, ist die Struktur freier wählbar, als bei einem Textdokument.

Als Basis bzw. Einstieg für den Leser sollte eine readme.md Datei existieren, welche eine Einleitung beschreibt und dann auf weitere Dateien bzw. Themen verweist.

Die erste Übersicht kann z.B. wie folgt strukturiert sein:

Einleitung

  • Ein kurzer Überblick über die Ziele der Software und deren Funktionsumfang.

Übersicht / Struktur der Dokumentation

  • Übersicht über die Struktur der Dokumentation mit Verweisen auf andere Bereiche und Dokumente.
    • Systemdokumentation
    • Erweiterungen (Bots)

6.1. Systemdokumentation

Eine Beschreibung der internen Arbeitsweise der Software, einschließlich der Schnittstellen und APIs Beschreibung der Code-Architektur und -Struktur Die folgenden Punkte sind als Ideenhilfe zu sehen (Basierend auf Arc42):

6.1.1. Einleitung

Eine kurze Einführung in das Dokument, einschließlich des Zwecks und des Kontexts der Softwarearchitektur.

6.1.2. Qualitätsziele (HIER NICHT ERFORDERLICH)

Eine Beschreibung der wichtigsten Qualitätsanforderungen an die Software, einschließlich Leistung, Sicherheit, Verfügbarkeit, Wartbarkeit und Skalierbarkeit.

6.1.3. Anforderungen (HIER NICHT ERFORDERLICH)

Eine Auflistung der funktionalen und nicht funktionalen Anforderungen an die Software.

6.1.4. Randbedingungen (HIER NICHT ERFORDERLICH)

Alles, was das Team beim Design und der Implementierung der Architektur einschränkt. Diese Einschränkungen sind manchmal auch außerhalb eines Projekts in der gesamten Organisation gültig.

Für eine tragfähige Architektur sollten alle genau wissen, wo Ihre Freiheitsgrade bezüglich der Entwurfsentscheidungen liegen und wo Sie Randbedingungen beachten müssen.

6.1.5. Kontextabgrenzung

Grenzt das System, an dem Sie arbeiten, von externen Kommunikationspartnern (Nachbarsystemen und Benutzern) ab. Zeigt die externen Schnittstellen aus Sicht des Business (immer) und aus Sicht der Technologie (optional)

Motivation

Die fachlichen und technischen Schnittstellen zur Kommunikation gehören zu den kritischsten Aspekten eines Systems. Stellen Sie sicher, dass Sie diese komplett verstanden haben.

Form

Verschiedene Optionen:

  • Diverse Kontextdiagramme
  • Listen von Kommunikationsbeziehungen mit deren Schnittstellen
  1. Fachlichter Kontext

    Festlegung aller Kommunikationsbeziehungen (Nutzer, IT-Systeme, …) mit Erklärung der fachlichen Ein- und Ausgabedaten oder Schnittstellen. Zusätzlich (bei Bedarf) fachliche Datenformate oder Protokolle der Kommunikation mit den Nachbarsystemen.

    Motivation

    Alle Beteiligten müssen verstehen, welche fachlichen Informationen mit der Umwelt ausgetauscht werdenw.

    Form

    Alle Diagrammarten, die das System als Blackbox darstellen und die fachlichen Schnittstellen zu den Nachbarsystemen beschreiben. Alternativ oder ergänzend können Sie eine Tabelle verwenden. Der Titel gibt den Namen Ihres Systems wieder; die drei Spalten sind: Kommunikationsbeziehung, Eingabe, Ausgabe.

  2. Technischer Kontext

    Technische Schnittstellen (Kanäle, Übertragungsmedien) zwischen dem System und seiner Umwelt. Zusätzlich eine Erklärung (mapping), welche fachlichen Ein- und Ausgaben über welche technischen Kanäle fließen.

    Motivation

    Viele Stakeholder treffen Architekturentscheidungen auf Basis der technischen Schnittstellen des Systems zu seinem Kontext. Insbesondere bei der Entwicklung von Infrastruktur oder Hardware sind diese technischen Schnittstellen durchaus entscheidend.

    Form

    Beispielsweise UML Deployment-Diagramme mit den Kanälen zu Nachbarsystemen, begleitet von einer Tabelle, die Kanäle auf Ein-/Ausgaben abbildet.

6.1.6. Lösungsstrategie

Kurzer Überblick über die grundlegenden Entscheidungen und Lösungsansätze, die Entwurf und Implementierung des Systems prägen. Hierzu gehören:

  • Technologieentscheidungen
  • Entscheidungen über die Top-Level-Zerlegung des Systems, beispielsweise die Verwendung gesamthaft prägender Entwurfs- oder Architekturmuster,
  • Entscheidungen zur Erreichung der wichtigsten Qualitätsanforderungen sowie
  • relevante organisatorische Entscheidungen, beispielsweise für bestimmte Entwicklungsprozesse oder Delegation bestimmter Aufgaben an andere Stakeholder.

Motivation

Diese wichtigen Entscheidungen bilden wesentliche „Eckpfeiler“ der Architektur. Von ihnen hängen viele weitere Entscheidungen oder Implementierungsregeln ab.

Form

Fassen Sie die zentralen Entwurfsentscheidungen kurz zusammen. Motivieren Sie, ausgehend von Aufgabenstellung, Qualitätszielen und Randbedingungen, was Sie entschieden haben und warum Sie so entschieden haben. Vermeiden Sie redundante Beschreibungen und verweisen Sie eher auf weitere Ausführungen in Folgeabschnitten.

6.1.7. Bausteinsicht

Statische Zerlegung des Systems. Die Abstraktion des Sourcecodes, dargestellt als Hierarchie von “White-Boxes” (die wiederum kleinere Black-Boxes beinhalten), bis zu einem angemessenen Detaillierungsgrad

Motivation

Behalten Sie den Überblick über den Quellcode, indem Sie die statische Struktur des Systems durch Abstraktion verständlich machen. Damit ermöglichen Sie Kommunikation auf abstrakterer Ebene, ohne zu viele Implemenierungsdetails offenlegen zu müssen.

Form

Die Bausteinsicht ist eine hierarchische Sammlung von Blackboxen und Whiteboxen und deren Beschreibungen.

6.1.8. Laufzeitsichten

Das Verhalten der Bausteine in Form von dynamischen Szenarien, die die wichtigsten Prozesse oder Features abdecken, Interaktionen an kritischen externen Schnittstellen oder “interessante” interne Abläufe und kritische Ausnahme- oder Fehlerfälle.

Motivation

Sie sollten verstehen, wie (Instanzen von) Bausteine(n) Ihres Systems ihre jeweiligen Aufgaben erfüllen und zur Laufzeit miteinander kommunizieren. Nutzen Sie diese Szenarien in der Dokumentation hauptsächlich für eine verständlichere Kommunikation mit denjenigen Stakeholdern, die die statischen Modelle (z.B. Bausteinsicht, Verteilungssicht) weniger verständlich finden.

Form

Für die Beschreibung von Szenarien gibt es zahlreiche Ausdrucksmöglichkeiten. Nutzen Sie beispielsweise:

  • Nummerierte Schrittfolgen oder Aufzählungen in Umgangssprache
  • Aktivitäts- oder Flussdiagramme
  • Sequenzdiagramme
  • BPMN (Geschäftsprozessmodell und -notation) oder EPKs (Ereignis-Prozessketten)
  • Zustandsautomaten
  • …

6.1.9. Infrastruktursicht

Technische Infrastruktur mit Prozessoren, Systemtopologie, und die Abbildung der Software-Bausteine auf diese Infrastruktur.

Motivation

Software läuft nicht ohne Infrastruktur. Diese zugrundeliegende Infrastruktur beeinflusst Ihr System und/oder querschnittliche Lösungskonzepte, daher müssen Sie diese Infrastruktur kennen.

Form

Das oberste Verteilungsdiagramm könnte bereits in Ihrem technischen Kontext enthalten sein, mit Ihrer Infrastruktur als EINE Blackbox. Jetzt zoomen Sie in diese Infrastruktur mit weiteren Verteilungsdiagrammen hinein:

  • Die UML stellt mit Verteilungsdiagrammen (Deployment diagrams) eine Diagrammart zur Verfügung, um diese Sicht auszudrücken. Nutzen Sie diese, evtl. auch geschachtelt, wenn Ihre Verteilungsstruktur es verlangt.
  • Falls Ihre Infrastruktur-Stakeholder andere Diagrammarten bevorzugen, die beispielsweise Prozessoren und Kanäle zeigen, sind diese hier ebenfalls einsetzbar.

6.1.10. Querschnittliche Konzepte

7. Hinweise

Dont Panik!

Das System scheint auf den ersten Blick komplex zu sein. Versuchen Sie das große Problem in kleinere Einheiten aufzuteilen und bearbeiten Sie zunächst diese kleineren Problemstellungen.

Versuchen Sie früh mittels Prototypen die externen Schnittstellen zu testen und eine Idee über die Ansprache und Verarbeitung dieser REST-APIs zu bekommen.

Denken Sie bei der Konzeption des Systems an die gelernten Prinzipen und allg. Grundsätze des Systementwurfs. Setzen Sie einen Fokus auf folgende SOLID Prinzipien:

  • Single Responsibility Principle: Jede Klasse sollte nur eine Verantwortlichkeit haben und für eine spezifische Aufgabe zuständig sein.
  • Open/Closed Principle: Klassen sollten für Erweiterungen offen, aber für Änderungen geschlossen sein. Das bedeutet, dass neue Funktionen hinzugefügt werden können, ohne den bestehenden Code zu ändern.
    • Liskov Substitution Principle: Objekte einer abgeleiteten Klasse sollten jederzeit durch Objekte der Basisklasse ersetzbar sein, ohne dass es zu Fehlern kommt.
  • Interface Segregation Principle: Klienten sollten nur von den Methoden abhängig sein, die sie wirklich benötigen, und nicht von unnötigen Methoden einer Schnittstelle.
  • Dependency Inversion Principle: Abhängigkeiten sollten von abstrakten Klassen oder Schnittstellen abhängen, und nicht von konkreten Implementierungen.

Bei der Umsetzung ist eine lose Kopplung von Modulen und Klassen von Vorteil. Eine lose Kopplung der Module und Klassen bedeutet, dass diese unabhängig voneinander existieren und Änderungen in einem Modul oder einer Klasse keine Auswirkungen auf andere haben sollten. Die Module sollten über klare Schnittstellen kommunizieren, die unabhängig voneinander getestet werden können.

Um diese Prinzipien in das Chatbot-System zu integrieren, können Sie folgende Schritte unternehmen:

  • Definieren Sie klare Schnittstellen für jedes Modul und jede relevante Klasse, um die Abhängigkeiten zu reduzieren.
  • Entwickeln Sie jede Klasse so, dass sie nur für eine spezifische Aufgabe verantwortlich ist und keine unnötigen Abhängigkeiten hat.
  • Verwenden Sie abstrakte Klassen oder Schnittstellen für Abhängigkeiten, um eine lose Kopplung zu gewährleisten.
  • Verwenden Sie das Open/Closed-Prinzip, um neue Chatbots einfach hinzuzufügen, ohne den bestehenden Code zu ändern.
  • Trenne Fachlichkeit und Technik um die Verantwortlichkeiten einzugrenzen und eine geringe Kopplung zu ermöglichen.
  • Trennen Sie die Datenbankzugriffslogik von anderen Modulen, um die Abhängigkeiten zu reduzieren und die Modularität zu erhöhen.

Durch die Einhaltung dieser Prinzipien kann das Chatbot-System skalierbar, erweiterbar und leicht zu warten sein, während die Code-Qualität und die Lesbarkeit erhöht werden.
